SST use cases

  • Use SST for coursework. Just run existing models and tweak available parameters. Semester-long coursework or project.
    • Performance and scaling are not crucial. Existing components suffice
    • May be focused on a specific aspect (network, memory, cpu) or generic system modeling
    • Desktop-scale workflow to Excel for plotting
  • Extension of components for graduate research
    • 2-3 years of actual research with a year of onboarding.
    • Performance and scaling are nice-to-have
    • Desktop-scale workflow to Excel for plotting
    • Graduate student starts by writing either their own serial DES of a specific aspect, e.g., network, memory, cpu and then migrate into SST for more complicated aspects of integration. System-level consequences of changing one aspect - memory size, network topology, routing strategy, CPU isa, CPU cache size, file system access
      Or
      How an application will perform on a system with specific memory size, network topology, routing strategy, CPU isa, CPU cache size, file system access
  • non-SST architect: Given a set of parameters for an existing model, produce plots.
    • SST is not that visible to user
    • Desktop-scale workflow to Excel for plotting
  • Core IP development for commercial companies
    • Modifying sst-core is typically seen as out-of-scope
    • Desktop-scale workflow to Excel for plotting; may grow to use an HPC to host SST simulations. At large scale the simulation logs output in hdf5 or elastic
    • Core IP may be narrowly focused on network, memory, cpu; or may be an integrator asking questions of the system design.
    • Limitations of performance and scaling constrain scope of business use case
    • The competition to using SST is just buying off the shelf DES, e.g. from Siemens
  • Folks interested in answering complex questions but unable to simulate
    • Interested in modifying sst-core
    • Includes both (desktop-scale workflow to Excel for plotting) and (large scale the simulation logs output in hdf5 or elastic)

For question about non-parallel applications, use Gem5

For network questions that doen't involve endpoints, use OMNet++ or NS2/NS3

  • Why is running my MPI application on a simulated HPC architecture non-trivial?
    • SST is just a DES. That means it doesn't run an operating system or have a compiler. An application needs to be compiled, and the compiled application relies on system calls provided by the operating system. Also, applications may rely on having access to a filesystem. There's no model of I/O included in SST. As a consequence, “running an application” is often replaced with relying on patterns of message passing rather than the actual messages associated with an application.
    • See QEMU and gem5 instead
